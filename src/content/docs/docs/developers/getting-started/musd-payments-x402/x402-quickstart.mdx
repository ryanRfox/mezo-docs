This quickstart runs top to bottom. By the end, you will have paid
`$0.001` in MUSD on Mezo Testnet against a live demo seller, then against
your own local seller — reaching the **Pay now** button successfully in
both cases.

## What you will build

A minimal two-part client/server app for purchasing a `/paid` API endpoint using MUSD:

- **Merchant Server**: an Express server with one protected route that returns
  JSON only after a valid MUSD payment.
- **Buyer Client**: a browser client that loads the bundled x402 paywall, connects
  a wallet, pays on Mezo Testnet, and receives the protected API response.

For the conceptual context of what x402 is, start with the [x402 Overview](../).

## Prerequisites

- **Node.js 20+** and **pnpm 9+** on your machine.
- A browser wallet extension that supports EVM networks (MetaMask, Rabby,
  Coinbase Wallet, or any wallet that exposes an EIP-1193 provider).
- **~20–30 minutes.** Getting testnet MUSD involves a faucet drip + a
  borrow transaction; run those early (Steps 1–2) and they'll be ready
  by the time you need them.

<Aside type="tip" title="Gas on Mezo with x402">
Neither the buyer nor the merchant needs a BTC balance to complete
x402 MUSD payments. The facilitator pays gas on behalf of both parties
and settles the transfer on chain. Only the buyer needs MUSD.
</Aside>

## Step 1: Add Mezo Testnet to your wallet

Open your wallet's networks settings and add a custom network with
these parameters:

| Field | Value |
|---|---|
| Network name | Mezo Testnet |
| Chain ID | `31611` |
| RPC URL | `https://rpc.test.mezo.org` |
| Block explorer | `https://explorer.test.mezo.org` |
| Currency symbol | BTC |

For mainnet parameters and alternative RPC providers, see
[Set Up Developer Environment](/docs/developers/getting-started/configure-environment).

## Step 2: Get testnet MUSD to pay with

MUSD on testnet is **minted, not faucet-dispensed** — you borrow it
against testnet BTC.

<Aside type="caution" title="One-time minimum borrow">
Borrow opens require a minimum debt of **2,000 MUSD** (1,800 principal +
200 MUSD gas deposit). This is a protocol floor, not a per-test cost —
you borrow once and reuse the balance across many paid requests. At
`$0.001` per call, a single borrow covers about two million test
payments.
</Aside>

<Steps>

1. Request testnet BTC from the [Mezo Faucet](https://faucet.test.mezo.org)
   or the `#testnet` channel in [Mezo Discord](https://discord.com/invite/mezo).
   Wait for the drip to land in your wallet.

2. Borrow MUSD at [mezo.org/feature/borrow](https://mezo.org/feature/borrow)
   with your wallet on Mezo Testnet. Full walkthrough with screenshots:
   [Borrow and Mint MUSD](/docs/users/musd/mint-musd/).

3. Confirm your MUSD balance. The testnet MUSD token contract is
   [`0x1189…Ac503`](https://explorer.test.mezo.org/token/0x118917a40FAF1CD7a13dB0Ef56C86De7973Ac503).
   Either paste your wallet address into that explorer page's search
   bar, or add `0x118917a40FAF1CD7a13dB0Ef56C86De7973Ac503` as a
   custom token in your wallet. Either way you should see at least
   1,800 MUSD (the principal from the previous step).

</Steps>

## Step 3: Smoke-test your wallet against the live x402 demo

Before installing any packages or writing any code, confirm that your
wallet, network, and MUSD balance are all ready by paying a live x402
seller. This isolates "did I set up my wallet correctly?" from "did I
wire my own server correctly?" — any trouble you hit here is a wallet
or MUSD issue, not a code issue.

<Steps>

1. Confirm the demo seller is up and paywall-gated:

   ```bash
   curl -s -o /dev/null -w "%{http_code}\n" https://humor.vativ.io/joke
   # Expect: 402
   ```

2. Open [`https://humor.vativ.io/joke`](https://humor.vativ.io/joke) in
   the same browser that holds your Mezo Testnet wallet.

3. The paywall loads. You should see:

   - A **Payment Required** heading
   - `please pay $0.001 Mezo USD`
   - A wallet-select dropdown and **Connect wallet** button

4. Select your wallet and connect it. Confirm you are on
   **Mezo Testnet (chain ID `31611`)** when the wallet prompts.

5. Click **Pay now**, then sign the payment authorization in your
   wallet when prompted.

6. The paywall retries the request with your payment. Once the
   facilitator settles on chain, the joke renders in the browser.

</Steps>

If the joke appears, your wallet setup is proven. Proceed to Step 4.

If you don't reach the joke, stop here and fix the wallet side before
writing any server code — see [Troubleshooting](#troubleshooting) for
common wallet/MUSD symptoms. A broken wallet at this step will look
identical to a broken server at Step 6; sort it out now.

## Step 4: Install x402 packages

Create a fresh project for your own seller:

```bash
mkdir mezo-x402-server && cd mezo-x402-server
pnpm init
pnpm add @x402/paywall @x402/evm @x402/core @x402/express express
pnpm add -D typescript tsx @types/express @types/node
```

<Aside type="note" title="About pnpm init's defaults">
`pnpm init` generates a `package.json` with `"main": "index.js"` and no
`"type": "module"`. Neither matters here — you will run `server.ts` via
`tsx`, which handles ESM and CommonJS without configuration.
</Aside>

<Aside type="note" title="Peer-dependency warnings are expected">
`pnpm` may print yellow peer-dependency warnings during install (for
example `Issues with peer dependencies found`, or notes about
`@x402/*` peer version ranges). These are informational and safe to
ignore for this quickstart — the install has succeeded as long as the
two `pnpm add` commands exit `0`. If a real error halts the install,
`pnpm` prints in red and exits non-zero.
</Aside>

## Step 5: Build your own seller

The seller is an Express server that uses `@x402/express` middleware to
gate a route behind an MUSD payment. The middleware wires together three
pieces: an `x402ResourceServer` that negotiates with a facilitator, an
`ExactEvmScheme` registered for Mezo Testnet, and a route config that
says what the endpoint costs.

You'll need a Mezo Testnet **receiving address** — any EVM address you
control (e.g., copy your MetaMask account address). It can be the
same wallet you used to borrow MUSD in Step 2, or a different one;
the merchant address never needs to hold BTC or MUSD, it only
receives payments.

Still inside `mezo-x402-server/` (from Step 4), create `server.ts`:

```typescript
import express, { type Request, type Response } from 'express';
import { paymentMiddleware } from '@x402/express';
import { x402ResourceServer, HTTPFacilitatorClient } from '@x402/core/server';
import { ExactEvmScheme } from '@x402/evm/exact/server';
import { createPaywall, evmPaywall } from '@x402/paywall';

const EVM_ADDRESS     = process.env.EVM_ADDRESS;         // Your Mezo Testnet receiving address (must include 0x prefix)
const FACILITATOR_URL = 'https://facilitator.vativ.io/'; // Mezo-capable facilitator provided by community

if (!EVM_ADDRESS) throw new Error('Set EVM_ADDRESS to your Mezo Testnet receiving address.');

const app = express();

const facilitatorClient = new HTTPFacilitatorClient({ url: FACILITATOR_URL });

// Bundled paywall UI: knows how to render MUSD at 18 decimals and connect EVM
// wallets. Without this, the middleware falls back to a stub that defaults to
// USDC formatting (6 decimals) and shows an "Install @x402/paywall" placeholder.
const paywall = createPaywall()
  .withNetwork(evmPaywall)
  .build();

app.use(
  paymentMiddleware(
    {
      'GET /paid': {
        accepts: [
          {
            scheme: 'exact',
            price: '$0.001',             // charged in MUSD; resolves via DEFAULT_STABLECOINS
            network: 'eip155:31611',     // Mezo Testnet, CAIP-2 string (not 'mezo-testnet')
            payTo: EVM_ADDRESS,
          },
        ],
        description: 'A paid endpoint on Mezo',
        mimeType: 'application/json',
      },
    },
    new x402ResourceServer(facilitatorClient).register(
      'eip155:*',                  // EVM-family scheme handler; route config above pins the actual chain
      new ExactEvmScheme(),
    ),
    undefined,    // paywallConfig (defaults are fine)
    paywall,      // ← the actual bundled UI provider
  ),
);

app.get('/paid', (req: Request, res: Response) => {
  res.json({
    message: 'Thank you for your payment.',
    servedAt: new Date().toISOString(),
  });
});

app.listen(3000, () => {
  console.log('seller listening on http://localhost:3000');
});
```

## Step 6: Run your seller and pay it yourself

<Aside type="caution" title="Check for a stale seller on :3000 first">
If any process is already bound to `localhost:3000` — a previous
`tsx server.ts` you forgot to stop, or anything else — the checks below
will pass because the **other** process answered, not yours. Payments
made in the browser then route to whatever `EVM_ADDRESS` that stale
server was configured with, not what you just set. This failure mode
looks identical to success; the only signal something is wrong is that
your terminal shows no request logs while someone else's does.

Kill any stale listener before continuing:

```bash
lsof -i :3000          # No output = port is free, skip the kill
kill $(lsof -ti :3000) # If the command above printed anything, run this
```

Alternatively, change `app.listen(3000, …)` in `server.ts` to a free
port.
</Aside>

Start the server with your receiving address:

```bash
EVM_ADDRESS=0xYourMezoTestnetAddress \
  pnpm exec tsx server.ts
```

You should see `seller listening on http://localhost:3000`. The
middleware will now intercept unpaid requests to `/paid`, respond with
HTTP `402 Payment Required`, and return a full HTML paywall page. No
client-side code is required.

<Steps>

1. Sanity-check that the server responds `402`:

   ```bash
   curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3000/paid
   # Expect: 402
   ```

   (If you want to inspect the paywall HTML, open the URL in a browser
   — the next step. `curl -i` works but prints the full paywall page
   to the terminal, which is a lot.)

2. Open `http://localhost:3000/paid` in the same browser you used for
   the live-demo smoke test in Step 3.

3. The paywall loads — same UI as Step 3, but now served by your own
   code.

4. Connect your wallet, confirm **Mezo Testnet (chain ID `31611`)**,
   click **Pay now**, and sign the authorization.

5. The paywall retries the request. Once the facilitator settles on
   chain, the JSON response body renders:

   ```json
   {
     "message": "Thank you for your payment.",
     "servedAt": "2026-..."
   }
   ```

</Steps>

If the browser console (DevTools → Console, usually **F12** or
**⌘⌥J** on macOS) logs `EVM Payment required initialized` with no
thrown errors, your install and middleware wiring are correct.

That's the full loop: you've just served a paywalled HTTP resource on
Mezo Testnet and paid for it from a wallet you control.

<Aside type="note" title="Source of the live demo">
The seller running at `humor.vativ.io` (used in Step 3) is a richer
reference — it also includes a non-browser `@x402/fetch` client and a
README covering end-to-end wallet funding. Source at
[`vativ/mezo-hack/apps/humor`](https://github.com/vativ/mezo-hack/tree/main/apps/humor).
Clone and adapt when you want to go beyond the snippets on this page.
</Aside>

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| `does not provide an export named 'DEFAULT_STABLECOINS'` at runtime | Installed `@x402/paywall`/`@x402/evm` are canonical `2.10.x`, which lacks Mezo support | Apply the [Mezo preview overrides](#mezo-preview-overrides) |
| Paywall shows `$10000000000.00` instead of `$0.001` | Same as above — canonical 2.10.x is formatting MUSD as 6-decimal USDC | Apply the [Mezo preview overrides](#mezo-preview-overrides) |
| `EADDRINUSE: address already in use :::3000` | Another process is bound to port 3000 | Run `lsof -i :3000` to find the holder, or change `app.listen(3000, …)` to a free port |
| `tsx: command not found` | `tsx` dev dep didn't install, or command is being run outside the project directory | Re-run `pnpm add -D tsx` inside the project; invoke as `pnpm exec tsx server.ts` |
| `SyntaxError` on `import` / TS syntax errors from Node | Using Node.js &lt; 20 | Upgrade to Node.js 20+ (`node --version` to check) |
| Wallet prompts to switch network / payment never confirms | Wallet is on the wrong Mezo network | Confirm the wallet is on the chain ID the seller expects (`31611` for Testnet, `31612` for Mainnet) |
| `402 Payment Required` returned but paywall UI never renders | `network` string is not a CAIP-2 identifier | Use `'eip155:31611'` or `'eip155:31612'` exactly; `'mezo-testnet'` and bare `31611` are both rejected |
| Facilitator returns `unsupported network` | `FACILITATOR_URL` in `server.ts` points at `x402.org/facilitator` (no Mezo support) | Set the constant to a Mezo-capable facilitator (`https://facilitator.vativ.io/` for Testnet) |
| Paywall loads but **Pay now** never settles, no error | Wallet has no MUSD, or MUSD is on the wrong network | Revisit Step 2; confirm the testnet MUSD token contract shows a balance |
| Step 3 demo (humor.vativ.io) never settles | Same wallet/MUSD/network issue — isolate here before proceeding to Step 4 | Do not skip the Step 3 smoke test; fix the wallet side before writing server code |

## Mezo preview overrides

You only need this section if `pnpm` / `npm` / `yarn` resolved
`@x402/paywall` at `2.10.x`. Canonical `2.11.0+` includes Mezo support
and works without any overrides.

Check what got installed:

```bash
pnpm list @x402/paywall
```

If the version is `2.11.0` or later: you can ignore this section. If it
is `2.10.x`: apply the overrides below.

### Add the overrides

Open `package.json` and add this block at the top level (same nesting
level as `dependencies`):

<Tabs>
  <TabItem label="pnpm">

```json
"pnpm": {
  "overrides": {
    "@x402/paywall": "https://github.com/vativ/x402-mezo-preview/releases/download/v2.10.0-mezo.6/x402-paywall-2.10.0-mezo.6.tgz",
    "@x402/evm":     "https://github.com/vativ/x402-mezo-preview/releases/download/v2.10.0-mezo.6/x402-evm-2.10.0-mezo.6.tgz"
  }
}
```

Then re-install:

```bash
pnpm install
```

  </TabItem>
  <TabItem label="npm">

Use npm's top-level `overrides` (not under a `pnpm` key):

```json
"overrides": {
  "@x402/paywall": "https://github.com/vativ/x402-mezo-preview/releases/download/v2.10.0-mezo.6/x402-paywall-2.10.0-mezo.6.tgz",
  "@x402/evm":     "https://github.com/vativ/x402-mezo-preview/releases/download/v2.10.0-mezo.6/x402-evm-2.10.0-mezo.6.tgz"
}
```

Then re-install:

```bash
npm install
```

  </TabItem>
  <TabItem label="yarn">

Use Yarn's `resolutions` block:

```json
"resolutions": {
  "@x402/paywall": "https://github.com/vativ/x402-mezo-preview/releases/download/v2.10.0-mezo.6/x402-paywall-2.10.0-mezo.6.tgz",
  "@x402/evm":     "https://github.com/vativ/x402-mezo-preview/releases/download/v2.10.0-mezo.6/x402-evm-2.10.0-mezo.6.tgz"
}
```

Then re-install:

```bash
yarn install
```

  </TabItem>
</Tabs>

### Verify

Both packages should now report `2.10.0-mezo.6`:

```bash
grep '"version"' node_modules/@x402/paywall/package.json node_modules/@x402/evm/package.json
```

If either still reads plain `2.10.0`, the override didn't apply. Check
that the overrides block sits at the top level of `package.json`
(alongside `dependencies`, not nested inside it) and re-run the install
command for your package manager.

<Aside type="caution" title="Override both packages together">
Preview `@x402/paywall` imports `DEFAULT_STABLECOINS` from preview
`@x402/evm`. If only one is overridden, the transitive canonical
`@x402/evm` surfaces as `does not provide an export named
'DEFAULT_STABLECOINS'` at runtime.
</Aside>
